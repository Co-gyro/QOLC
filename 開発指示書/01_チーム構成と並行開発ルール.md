**新規プロジェクトの作成は不要です。**

## チーム構成（Claude Code 5セッション並行体制）

| チーム | 担当領域 | Git ブランチ規則 |
|---|---|---|
| **Team A: 基盤** | DB設計、認証、ミドルウェア | `feat/foundation-*` |
| **Team B: 決済** | USEN PSP API連携、決済フロー | `feat/payment-*` |
| **Team C: ポータルUI** | 4ポータルの画面実装 | `feat/portal-*` |
| **Team D: データ処理** | CSV変換、レセプト解析、PDF生成 | `feat/data-*` |
| **Team E: 品質・セキュリティ** | テスト、セキュリティレビュー、CI/CD | `feat/quality-*` |

## 作業の依存関係（実行順序）

```
Phase 1（Week 1-2）: Team A（基盤）が先行
  ├── DB マイグレーション作成
  ├── Supabase Auth 設定
  ├── middleware.ts（認証・ルーティング）
  └── RLS ポリシー設計

Phase 2（Week 2-3）: Team B + C + D が並行開始（Team A の基盤を利用）
  ├── Team B: USEN PSP API クライアント実装
  ├── Team C: 各ポータルのレイアウト・画面
  └── Team D: CSV/レセプト/PDF処理ロジック

Phase 3（Week 3-4）: 統合 + Team E がレビュー
  ├── Team C が Team B の決済APIを画面に統合
  ├── Team D のデータ処理を管理画面に統合
  └── Team E: セキュリティレビュー + E2Eテスト

Phase 4（Week 4-5）: 全体テスト + 修正
  └── Team E 主導で品質ゲート通過確認
```

## Git 運用ルール

### ブランチ戦略
```
main          ← 本番デプロイ（直接pushは禁止）
  └── develop ← 統合ブランチ（各チームがここにマージ）
       ├── feat/foundation-db-schema
       ├── feat/foundation-auth
       ├── feat/payment-hmac
       ├── feat/payment-member-api
       ├── feat/portal-admin-layout
       ├── feat/portal-facility-residents
       ├── feat/data-receipt-parser
       └── feat/quality-e2e-setup
```

### マージルール
1. 各チームは自分の `feat/*` ブランチで作業
2. 完了したら `develop` へのマージ前に **Team E がレビュー**
3. `develop` → `main` のマージは **全テスト通過 + セキュリティチェック通過** が条件

### コンフリクト防止
- **各チームの担当ディレクトリを明確に分離**（下記参照）
- 共通ファイル（types.ts, middleware.ts等）の変更は Team A のみが行う
- 他チームが共通ファイルの変更が必要な場合は、Team A に依頼する

### チームごとの担当ディレクトリ

| チーム | 専有ディレクトリ（他チームは変更禁止） |
|---|---|
| Team A | `supabase/`, `src/lib/supabase/`, `src/middleware.ts`, `src/types/` |
| Team B | `src/lib/payment/`, `src/app/api/payment/` |
| Team C | `src/app/admin/`, `src/app/facility/`, `src/app/provider/`, `src/app/user/`, `src/components/` |
| Team D | `src/lib/csv/`, `src/lib/pdf/`, `src/lib/receipt/`, `src/app/api/upload/` |
| Team E | `tests/`, `docs/security/`, `.github/`, `scripts/` |

## 品質ゲート（マージ前の必須チェック）

### Level 1: 各チームのセルフチェック（毎回）
- [ ] TypeScript コンパイルエラー 0件（`npx tsc --noEmit`）
- [ ] ESLint エラー 0件（`npx eslint src/`）
- [ ] ユニットテスト全通過（`npx vitest run`）
- [ ] 新規ファイルにテストがある

### Level 2: Team E レビュー（develop マージ前）
- [ ] セキュリティチェックリスト通過（別紙参照）
- [ ] RLS ポリシーのテスト通過
- [ ] 入力バリデーション（zod）の確認
- [ ] エラーハンドリングの確認（機密情報がエラーメッセージに含まれないか）
- [ ] E2Eテスト通過

### Level 3: リリースゲート（main マージ前）
- [ ] 全テストスイート通過
- [ ] セキュリティスキャン通過（npm audit, dependency check）
- [ ] USEN PSP テスト環境での決済フロー通過
- [ ] パフォーマンステスト（Lighthouse スコア 80+）
- [ ] アクセシビリティチェック（axe-core エラー 0件）
