# QOLC アップロードフォーマット管理

> 明細CSV/レセプトファイルの変換設計

---

## 1. 背景・課題

サービス提供者がアップロードするファイルには、以下の課題がある。

| 課題 | 内容 |
|------|------|
| レセプトシステムの多様性 | 各社（〇〇システム、△△システム等）でCSVフォーマットが異なる |
| レセプトがない事業者 | タクシー、買い物代行等は独自の請求書フォーマット |
| 新規フォーマット対応 | 新しい加盟店が増えるたびに対応が必要 |
| 文字コードの違い | UTF-8、Shift_JIS等が混在 |

### 解決策

運営者がフォーマット定義を登録・管理し、加盟店ごとに使用フォーマットを設定。
アップロード時に自動でQOLC統一フォーマットに変換する。

---

## 2. 全体の仕組み

### フォーマット管理の構成

```
【フォーマット管理（運営者）】
├── フォーマットA（〇〇レセプトシステム用）
├── フォーマットB（△△レセプトシステム用）
├── フォーマットC（タクシー請求書用）
└── フォーマットD（汎用CSV）

【加盟店登録時】
└── 使用フォーマットを選択（フォーマットA等）

【明細アップロード時】
└── 加盟店に紐づくフォーマットで自動変換 → QOLC統一フォーマットへ
```

### 処理フロー

| ステップ | 処理 | 実行者 |
|---------|------|--------|
| 1 | CSVファイルをアップロード | サービス提供者 |
| 2 | 加盟店のフォーマット設定を参照 | システム |
| 3 | カラムマッピングに従って変換 | システム |
| 4 | QOLC統一フォーマットに変換完了 | システム |
| 5 | 変換結果を確認画面で表示 | サービス提供者 |
| 6 | 確認後、決済処理へ | システム |

## 3. フォーマット定義の項目

### フォーマット基本情報

| 項目 | 内容 | 例 |
|------|------|-----|
| フォーマット名 | 識別用の名前 | 〇〇レセプトシステム v2 |
| フォーマットコード | システム内部の識別子 | FORMAT_001 |
| 説明 | フォーマットの説明 | 〇〇社のレセプトシステムから出力されるCSV |
| ファイル形式 | 対応するファイル形式 | CSV / Excel |
| 文字コード | ファイルの文字コード | UTF-8 / Shift_JIS |
| 区切り文字 | CSVの区切り文字 | カンマ / タブ |
| ヘッダー有無 | 1行目がヘッダーか | あり / なし |
| ステータス | 利用可否 | 有効 / 無効 |

### QOLC統一フォーマット（変換先）

すべてのアップロードファイルは、以下の統一フォーマットに変換される。

| カラム名 | 型 | 必須 | 説明 |
|---------|-----|------|------|
| resident_id | 文字列 | ○ | 入居者ID（施設内の識別子） |
| resident_name | 文字列 | ○ | 入居者氏名 |
| service_date | 日付 | ○ | サービス提供日 |
| service_type | 文字列 | ○ | サービス種別 |
| item_description | 文字列 | ○ | 明細項目の説明 |
| unit_count | 数値 | − | 単位数（点数） |
| unit_price | 数値 | − | 単価 |
| amount | 数値 | ○ | 金額 |
| insurance_amount | 数値 | − | 保険適用額 |
| self_pay_amount | 数値 | ○ | 自己負担額 |
| remarks | 文字列 | − | 備考 |

## 4. カラムマッピング設定

### カラムマッピングとは

元ファイルの各列を、QOLC統一フォーマットのどの列に対応させるかを定義する設定。

### マッピング例：〇〇レセプトシステム

| 元ファイルの列名 | → | QOLC統一フォーマット |
|----------------|---|---------------------|
| 利用者番号 | → | resident_id |
| 利用者氏名 | → | resident_name |
| サービス提供年月日 | → | service_date |
| サービス内容 | → | service_type |
| 摘要 | → | item_description |
| 単位数 | → | unit_count |
| 請求額 | → | amount |
| 保険給付額 | → | insurance_amount |
| 利用者負担額 | → | self_pay_amount |

### マッピング例：タクシー請求書

| 元ファイルの列名 | → | QOLC統一フォーマット |
|----------------|---|---------------------|
| お客様ID | → | resident_id |
| お客様名 | → | resident_name |
| 乗車日 | → | service_date |
| （固定値） | → | service_type = 'タクシー' |
| 行先 | → | item_description |
| 運賃 | → | amount |
| 運賃 | → | self_pay_amount（同じ値） |

### マッピング設定の機能

| 機能 | 説明 |
|------|------|
| 列の対応付け | 元ファイルの列 → 統一フォーマットの列 |
| 固定値の設定 | 特定のカラムに固定値を入れる（例：service_type = 'タクシー'） |
| 日付フォーマット変換 | 元ファイルの日付形式を指定（例：YYYY/MM/DD → YYYY-MM-DD） |
| 数値変換 | カンマ区切り等を数値に変換 |
| 列の結合 | 複数列を1つに結合（例：姓 + 名 → 氏名） |
| 列のスキップ | 不要な列を無視 |

## 7. エラー処理

アップロード時に発生しうるエラーと対処法。

| エラー種別 | 原因 | 対処法 |
|-----------|------|--------|
| 文字コードエラー | 設定と異なる文字コード | 正しい文字コードに設定変更 |
| カラム数不一致 | 列数が想定と異なる | フォーマット設定を確認・修正 |
| 必須項目欠落 | 必須カラムにデータがない | 元ファイルを修正して再アップロード |
| 日付形式エラー | 日付が想定形式でない | 日付フォーマット設定を修正 |
| 数値変換エラー | 数値に変換できない値 | 元ファイルを修正、または変換ルール追加 |
| 入居者ID不一致 | 該当する入居者が見つからない | 入居者マスタを確認、または手動マッピング |

### エラー時の画面表示

- エラー行をハイライト表示
- エラー内容を明示（「3行目：日付形式が不正です」等）
- エラー行をスキップして続行するオプション
- CSVでエラー一覧をダウンロード可能

## 8. データモデル

### UploadFormat（アップロードフォーマット）

```typescript
interface UploadFormat {
  id: string;
  name: string;                    // フォーマット名
  code: string;                    // フォーマットコード
  description?: string;            // 説明
  fileType: 'csv' | 'excel';       // ファイル形式
  encoding: 'utf-8' | 'shift_jis'; // 文字コード
  delimiter: ',' | '\t';           // 区切り文字
  hasHeader: boolean;              // ヘッダー有無
  columnMappings: ColumnMapping[]; // カラムマッピング
  status: 'active' | 'inactive';   // ステータス
  createdAt: Date;
  updatedAt: Date;
}

interface ColumnMapping {
  sourceColumn: string | number;   // 元ファイルの列（名前 or インデックス）
  targetColumn: string;            // QOLC統一フォーマットの列名
  transformType?: 'fixed' | 'date' | 'number' | 'concat';  // 変換タイプ
  transformConfig?: {              // 変換設定
    fixedValue?: string;           // 固定値
    dateFormat?: string;           // 日付フォーマット
    concatColumns?: string[];      // 結合する列
  };
}
```

## 9. まとめ

### フォーマット管理のメリット

| メリット | 説明 |
|---------|------|
| 柔軟性 | 新しいフォーマットに運営側で対応可能（コード変更不要） |
| 加盟店の負担軽減 | 既存のシステムからそのまま出力すればOK |
| 開発コスト削減 | 新フォーマット対応に開発者の作業が不要 |
| エラー防止 | マッピング設定で変換ルールを明確化 |
| トレーサビリティ | どのフォーマットで変換したか記録が残る |

### 運営者の役割への追加

運営者の役割 #14 として「アップロードフォーマット管理」を追加。

- 新規フォーマットの登録・設定
- カラムマッピングの設定・メンテナンス
- 変換エラー時の対応・調整
